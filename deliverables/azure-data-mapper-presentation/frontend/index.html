01

What the feature really is

Data Mapper is a project-scoped authoring pipeline inside Logic Apps Standard, not just a visual canvas.

Schemas
Visual map (.lml)
Generated .xslt
Workflow action

Best-fit use cases

B2B orders, invoices, canonical-model transforms, partner payload reshaping, and workflow-local schema-driven mappings.

Authoring model

Microsoft documents VS Code authoring with saved output in Artifacts/MapDefinitions and Artifacts/Maps.

Key tradeoff

The visual source and runtime source are separate artifacts, so teams must review both and treat mapper saves carefully.

Operational limit

Current mapper authoring is Standard-only and Windows-only, which matters immediately for team rollout planning.

02

XSLT 3.0, XSLT 1.0, and CDATA

The docs, the standard, and the public issue stream point to coexistence, not a full replacement story.

XSLT 3.0 in Logic Apps

  • Microsoft documents map type support for XSLT, XSLT 2.0, and XSLT 3.0.
  • Custom XML functions are documented in XSLT 3.0 style.
  • Advanced logic can live in project-scoped extension folders.

Compatibility with XSLT 1.0

  • Legacy XSLT maps still fit Logic Apps through Transform XML.
  • Data Mapper adds a different action path and a new .lml authoring artifact.
  • The safest modernization pattern is coexistence, not blanket migration.

Critical source-of-truth rule

Treat .lml as the authoring source and generated .xslt as the runtime source. If anyone hand-edits the generated XSLT, document that clearly because a later save from the mapper can overwrite it.
