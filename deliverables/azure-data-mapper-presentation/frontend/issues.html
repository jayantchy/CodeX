Known friction points

Use this page as a triage checklist before you assume the map logic is wrong. Several local testing and authoring behaviors are already documented or publicly reported.

Microsoft documented limitations

Authoring surface

Data Mapper authoring is documented for Visual Studio Code and Standard logic app projects, not Consumption projects.

Product scope

Operating system

Microsoft Learn currently lists Data Mapper as working only in VS Code on Windows.

Team setup risk

Filter behavior

The Filter function is documented as inconsistent for string comparisons while numeric conditions are called out as working when quoted.

Practical mitigations

Pin versions

Keep the VS Code extension version consistent during a sprint. Move everyone together after testing.

Save before testing

Local test results are only meaningful after the current .xslt has been regenerated.

Use sample fixtures

Keep representative JSON/XML samples in source control so regressions are visible in pull requests.
