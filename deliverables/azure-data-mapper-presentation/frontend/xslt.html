XSLT Compatibility

From XSLT 1.0 to 3.0

Azure Logic Apps Standard uses the Saxon.NET engine, meaning the Data Mapper compiles everything down to standard, highly compatible XSLT 3.0. Here is how traditional XSLT functionality maps to the visual designer and extensions.

How it's easily done with Data Mapper

Before writing raw XSLT, understand how Data Mapper visually replaces standard XSLT instructions. Data Mapper handles the heavy lifting of namespaces, looping, and document structure automatically.

Loops (xsl:for-each)

Data Mapper: Simply draw a line from a repeating source node (Array/List) to a repeating target node. Data Mapper automatically generates the xsl:for-each context loop.

Conditionals (xsl:if, xsl:choose)

Data Mapper: Use the If or Filter functions from the visual toolbox. Drag conditions onto target fields to control output logic without writing xsl:when blocks.

Value Mapping (xsl:value-of)

Data Mapper: Draw a direct line from a source field to a target field. Data Mapper generates the <xsl:value-of select="..."/> automatically behind the scenes.

Variables (xsl:variable)

Data Mapper: A single function node with multiple outputs visually represents a variable. Compute the value once (e.g., concatenate strings), and connect its output to multiple destinations.

Data Mapper Extensions Explained

When visual connections aren't enough, Data Mapper provides targeted ways to inject raw XSLT. These extensions bridge the gap between visual ease and XSLT 1.0–3.0 power.

Custom XML Functions

What it is: A separate `.xml` file containing standard xsl:function definitions. Placed in Artifacts/DataMapper/Extensions/Functions/.
How it compares: XSLT 1.0 lacked user-defined functions (relying on named templates). XSLT 2.0/3.0 introduced xsl:function. Data Mapper exposes these as custom, draggable blocks in the UI.
Use case: Reusable math, date parsing, or lookup logic.

Inline XSLT Snippets

What it is: Targeted `.xslt` files placed in Artifacts/DataMapper/Extensions/InlineXslt/, invoked via the visual Run XSLT function.
How it compares: Allows you to inject powerful XSLT 3.0 logic—like xsl:apply-templates for recursion, or complex xsl:for-each-group patterns—directly into a specific part of your visual map.
Use case: Advanced grouping, recursion, or migrating legacy BizTalk snippets.

XSLT Element Compatibility Matrix

An exhaustive review of standard XSLT elements and how they operate within the Azure Data Mapper context (generated maps, Inline XSLT, or Custom Functions).

XSLT Element XSLT Version Support Data Mapper Behavior
xsl:stylesheet / xsl:transform 1.0, 2.0, 3.0 Auto-Generated: Data Mapper creates the root element automatically, setting version="3.0".
xsl:output 1.0, 2.0, 3.0 Auto-Generated: Managed by Data Mapper (typically method="xml" or json depending on target schema).
xsl:template 1.0, 2.0, 3.0 Auto/Inline: Visual maps use a single root match="/". For custom templates, use Inline XSLT.
xsl:apply-templates 1.0, 2.0, 3.0 Inline XSLT Only: Visual mapper relies on explicit loops. Use Inline XSLT for recursive template matching.
xsl:value-of 1.0, 2.0, 3.0 Auto-Generated: Created simply by connecting a source node to a target node.
xsl:copy-of 1.0, 2.0, 3.0 Auto/Inline: Can be achieved visually by mapping parent nodes, but deeper control requires Inline XSLT.
xsl:choose, xsl:when, xsl:otherwise 1.0, 2.0, 3.0 Auto-Generated: Generated by the visual If/Else functions.
xsl:if 1.0, 2.0, 3.0 Auto-Generated: Generated by using the visual Filter or logical functions.
xsl:for-each 1.0, 2.0, 3.0 Auto-Generated: Automatically applied when connecting repeating source arrays to repeating target arrays.
xsl:variable 1.0, 2.0, 3.0 Auto/Inline: Function outputs act as variables visually. Explicit variables require Custom Functions or Inline XSLT.
xsl:param, xsl:with-param 1.0, 2.0, 3.0 Custom/Inline: Fully supported inside Custom XML Functions and Inline XSLT snippets.
xsl:function 2.0, 3.0 Custom Functions Only: Define these in a `.xml` file to expose them as visual blocks in the toolbox.
xpath functions/nodes/attributes 1.0, 2.0, 3.1 Supported: Use the visual Execute XPath function to write raw XPath 3.1 queries against the payload.
xsl:sort 1.0, 2.0, 3.0 Inline XSLT Only: Visual mapper does not have a native "Sort" block. Use Inline XSLT.
xsl:key 1.0, 2.0, 3.0 Inline XSLT Only: The Muenchian grouping method (1.0) or fast lookups (2.0/3.0) must be written via Inline XSLT.
xsl:for-each-group 2.0, 3.0 Inline XSLT Only: XSLT 2.0+ native grouping. Must be written via Inline XSLT (highly recommended over xsl:key grouping).
xsl:include, xsl:import 1.0, 2.0, 3.0 Managed: Data Mapper manages imports for Custom Functions automatically.
xsl:strip-space, xsl:preserve-space 1.0, 2.0, 3.0 Inline XSLT Only: White-space handling instructions must be provided via custom stylesheets.
xsl:namespace-alias, namespace-uri 1.0, 2.0, 3.0 Auto-Generated: Data Mapper resolves target schemas and handles namespace injection automatically.
xsl:attribute, xsl:element 1.0, 2.0, 3.0 Auto-Generated: Visual target mapping to an XML attribute generates xsl:attribute automatically.
xsl:decimal-format 1.0, 2.0, 3.0 Custom/Inline: If specialized number formatting is required, wrap it in a Custom XML Function.
xsl:number, xsl:text 1.0, 2.0, 3.0 Auto/Inline: Text injection is handled visually via string functions. Number sequencing requires Inline XSLT.
xsl:message, xsl:fallback, xsl:processing-instruction, xsl:comment 1.0, 2.0, 3.0 Inline XSLT Only: Low-level XML document manipulations require manual XSLT.
xsl:use-attribute-sets 1.0, 2.0, 3.0 Inline XSLT Only: Defining and applying reusable attribute sets requires custom code.
⚡ Azure Logic Apps Saxon.NET fully supports W3C XSLT 3.0 specifications.
