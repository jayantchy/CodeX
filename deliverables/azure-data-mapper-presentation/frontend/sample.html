A small order mapping

This example keeps the idea deliberately small: normalize a partner order payload into a canonical order shape, then test the map before adding it to the workflow.

Source payload

{
  "partnerOrder": {
    "id": "PO-1042",
    "orderedOn": "20260608",
    "buyer": { "name": "Contoso Retail" },
    "items": [
      { "sku": "A-100", "qty": 2, "price": 19.5 }
    ]
  }
}

Target payload

{
  "orderId": "PO-1042",
  "orderDate": "2026-06-08",
  "customerName": "Contoso Retail",
  "lines": [
    { "itemCode": "A-100", "quantity": 2, "lineTotal": 39 }
  ]
}

Mapping notes

Direct copy

id becomes orderId, and buyer.name becomes customerName.

Date normalization

Format YYYYMMDD into an ISO-style date string before downstream systems consume it.

Repeating lines

Map parent arrays and child item fields explicitly so loops are visible and testable.

XSLT 3.0 style helper

Data Mapper supports XSLT artifacts and custom XML functions that follow an XSLT 3.0 style. Use this for transformations that benefit from reusable logic.

<xsl:function name="dm:iso-date" as="xs:string">
  <xsl:param name="dateText" as="xs:string" />
  <xsl:sequence select="
    concat(
      substring($dateText, 1, 4), '-',
      substring($dateText, 5, 2), '-',
      substring($dateText, 7, 2)
    )
  " />
</xsl:function>
Save before local testing so the generated XSLT reflects the current map design.
