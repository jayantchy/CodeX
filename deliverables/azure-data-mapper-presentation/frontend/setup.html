Setup

Make local projects portable

A team setup should let any developer clone the repo, restore local settings, open the Logic App Standard project root, and edit maps without machine-specific drift.

Developer machine baseline

1. Pin tools

Agree on VS Code, Azure Logic Apps Standard extension, Azure Functions Core Tools, and Node/runtime versions for the team.

2. Open the root

Open the Standard logic app project root, where host.json, connections.json, and workflow folders live.

3. Restore local settings

Each developer creates their own root local.settings.json and, when needed, the matching workflow-designtime/local.settings.json.

Local settings template

Use the same shape for the project root and the workflow-designtime folder. Keep real subscription, resource group, storage, and connector values local to each developer.

The JSON below is valid. The designer copy needs the same comma-separated properties; missing commas will stop the local runtime from reading the file.
{
  "IsEncrypted": false,
  "Values": {
    "AzureWebJobsStorage": "UseDevelopmentStorage=true",
    "AzureWebJobsFeatureFlag": "EnableMultipleLanguageWorker",
    "FUNCTIONS_EXTENSION_VERSION": "~4",
    "FUNCTIONS_WORKER_RUNTIME": "dotnet",
    "WORKFLOWS_SUBSCRIPTION_ID": "",
    "WORKFLOWS_RESOURCE_GROUP_NAME": "",
    "WEBSITE_NODE_DEFAULT_VERSION": "~18",
    "APP_KIND": "workflowapp"
  }
}

Commit and ignore rules

Commit the artifacts that define the map and workflow. Ignore files that are secrets, machine state, or per-user editor preferences.

CommitIgnore
Artifacts/MapDefinitions/*.lmllocal.settings.json
Artifacts/Maps/*.xslt**/local.settings.json
Artifacts/Schemas/*.vscode/settings.json if personal
workflow.json, connections.json, host.jsonRaw keys, connection strings, and generated clutter

Recommended handoff rule

Commit a sanitized local.settings.sample.json with placeholders, then ask each developer to create their real local files from that template.

For Data Mapper work, also commit Artifacts/MapDefinitions, Artifacts/Maps, Artifacts/Schemas, and any Artifacts/DataMapper/Extensions files.

Best team handoff model

For normal collaboration, do not pass around exported zip files. Use Git as the source of truth for the VS Code workspace, and reserve export/package flows for bootstrapping from Azure or recovering a portal-created app.

Recommended repo shape

Keep the Logic App Standard project and custom code project as sibling folders in one repo or one VS Code .code-workspace. For the screenshot, that means TestSTdLA and FnStdLA stay side by side.

Why this avoids drift

Developers clone one repo, open the workspace root, restore local-only settings from samples, and let CI/CD deploy the same committed project files. No one depends on another machine's designer cache.

Transfer to another developer

1. Clone and open workspace

Clone the repo, then open the repo root or the checked-in .code-workspace. Do not open only Stateful1, Artifacts, or the function project folder.

2. Install the same tools

Use a committed .vscode/extensions.json to recommend Azure Logic Apps Standard, Azure Functions, C#, and Azurite extensions. Pin team runtime expectations in the README.

3. Create local settings

Copy local.settings.sample.json into both the project root and workflow-designtime as needed, then fill developer-specific subscription, resource group, storage, and connector values.

4. Restore and build code

Restore the FnStdLA project and build it locally. Microsoft's custom-code guidance expects code and workflow projects to remain separate folders.

5. Rebind cloud connectors

Sign in to Azure from VS Code, validate connections.json, and reselect managed API connections if the designer prompts for subscription or resource group context.

6. Smoke test the map

Open Data Mapper, load the existing map, save once if the extension regenerates XSLT, then run a small local workflow test before pushing changes.

Package vs Git vs deploy

Best for daily work

Use Git. It preserves history, review comments, branches, and the exact set of committed Logic App, Data Mapper, and custom code files.

Best for first import

Use Export logic app or Create new logic app workspace from package when the source currently lives in Azure or arrives as a zip package.

Best for Azure release

Use CI/CD or VS Code Deploy to logic app after local files are clean. Replace environment values through app settings, parameters, access policies, and connection scripts.

Practical rule: package import is a migration tool; Git is the collaboration tool; CI/CD is the promotion tool.

Pull request checks

Map review

Review the .lml design intent and the generated .xslt together. A visual save can regenerate runtime code.

Artifact presence

Confirm schemas, map definitions, maps, workflow folders, host.json, and connections.json are present before packaging.
