Setup & Deploy

Make local projects portable

A team setup should let any developer clone the repo, restore local settings, open the Logic App Standard project root, edit maps without machine-specific drift, and deploy directly to Azure from VS Code.

Developer machine baseline

1. Pin tools

Agree on VS Code, Azure Logic Apps Standard extension, Azure Functions Core Tools, and Node/runtime versions for the team.

2. Open the root

Open the Standard logic app project root, where host.json, connections.json, and workflow folders live.

3. Restore local settings

Each developer creates their own root local.settings.json and, when needed, the matching workflow-designtime/local.settings.json.

Making the workspace shareable

The goal: any developer clones the repo, runs three setup steps, and has a working Data Mapper environment with all connections and visual maps intact. No zip files, no screenshots of settings, no "works on my machine".

1. Use a .code-workspace file

Commit a VS Code workspace file that includes both the Logic App project and the custom code project as sibling folders. This ensures everyone opens the same root.

{
  "folders": [
    { "path": "MyLogicApp" },
    { "path": "MyCustomCode" }
  ],
  "settings": {
    "azureLogicAppsStandard.autoRuntimeDependenciesValidation": true
  }
}

2. Commit extensions.json

Pin the required extensions so VS Code prompts new developers to install them.

{
  "recommendations": [
    "ms-azuretools.vscode-azurelogicapps",
    "ms-azuretools.vscode-azurefunctions",
    "ms-dotnettools.csharp",
    "Azurite.azurite"
  ]
}

Save at .vscode/extensions.json.

3. Use local.settings.sample.json

Commit a sanitized template with placeholders. Each developer copies this to local.settings.json and fills in their own values. Never commit the real file.

4. Preserve connections visually

Commit connections.json with parameterized references. When a developer clones and opens the workspace, VS Code reads this file and restores the designer connections visually without manual re-creation.

5. Commit all Data Mapper artifacts

The map definition (.lml), generated XSLT (.xslt), schemas, and extension functions must all be in Git. Missing any one breaks the visual editor or runtime.

6. Pin the extension version

Current public issues suggest version drift causes real breakage. Document the exact extension version in the README and freeze it until the team tests the next release together.

Local settings template

Use the same shape for the project root and the workflow-designtime folder. Keep real subscription, resource group, storage, and connector values local to each developer.

The JSON below is valid. The designer copy needs the same comma-separated properties; missing commas will stop the local runtime from reading the file.
{
  "IsEncrypted": false,
  "Values": {
    "AzureWebJobsStorage": "UseDevelopmentStorage=true",
    "AzureWebJobsFeatureFlag": "EnableMultipleLanguageWorker",
    "FUNCTIONS_EXTENSION_VERSION": "~4",
    "FUNCTIONS_WORKER_RUNTIME": "dotnet-isolated",
    "WORKFLOWS_SUBSCRIPTION_ID": "",
    "WORKFLOWS_RESOURCE_GROUP_NAME": "",
    "WEBSITE_NODE_DEFAULT_VERSION": "~18",
    "APP_KIND": "workflowapp"
  }
}

Syncing workspace across developer machines

The trickiest part of sharing a Standard Logic App project is preserving connections and visual state. Here's how to ensure nothing gets lost.

What preserves connections

connections.json stores the connector definitions, authentication metadata, and runtime references. When committed to Git, any developer who clones and opens the project root will see the connectors in the designer — the visual links stay intact.

The key properties to watch:

  • managedApiConnections — API connector references
  • serviceProviderConnections — built-in connectors
  • connectionId — must point to a valid Azure resource

What can break across machines

  • Missing local.settings.json — designer won't start without required app settings
  • Missing workflow-designtime/local.settings.json — designer loses connection context
  • Wrong VS Code folder opened — extension doesn't recognize the project
  • Extension version mismatch — regenerated XSLT may differ
  • Missing schemas — Data Mapper can't render the visual map
  • Expired or missing Azure credentials — managed API connectors need re-auth
Golden rule: If a developer clones the repo, creates local.settings.json from the sample, signs into Azure in VS Code, and opens the workspace file — everything should just work. If it doesn't, something is missing from Git.

Commit and ignore rules

Commit the artifacts that define the map and workflow. Ignore files that are secrets, machine state, or per-user editor preferences.

Direct Deployment to Azure from VS Code

For smaller teams or fast prototyping, you can deploy your local workspace directly to Azure without setting up a CI/CD pipeline. The Azure Logic Apps extension handles packaging and publishing.

1. Prepare your local workspace

Ensure all maps are saved, XSLT is generated, and your local project is running successfully. Review your connections.json to ensure no local-only test connections will break in the cloud.

2. Open the Azure extension

Click the Azure icon in the Activity Bar. Under the Workspace section, ensure your Logic App project is recognized.

3. Deploy to Logic App

Click the Deploy to Logic App icon (or right-click the project folder and select the option). Follow the prompts to select your Azure Subscription and the target Logic App (Standard) resource.

4. Configure Cloud App Settings

Your local.settings.json is not deployed. You must manually configure any required settings (like connection string references or custom environment variables) in the Azure Portal under your Logic App's Configuration blade.

Environment Drift: If you use direct deployment, ensure your team communicates about cloud configuration changes. Without Infrastructure as Code (IaC) or CI/CD, connection strings or settings added via the portal are easily overwritten or forgotten during the next manual deploy.

New developer onboarding checklist

1. Clone and open workspace

Clone the repo, then open the .code-workspace file. Do not open Artifacts or a workflow subfolder directly.

2. Install recommended extensions

VS Code will prompt from .vscode/extensions.json. Accept all recommendations: Logic Apps Standard, Azure Functions, C#, Azurite.

3. Create local settings

Copy local.settings.sample.json into both the project root and workflow-designtime/. Fill in your Azure subscription, resource group, and storage values.

4. Start Azurite

Run Azurite for local storage emulation. The runtime needs it when AzureWebJobsStorage is set to UseDevelopmentStorage=true.

5. Sign into Azure

Sign in via the Azure extension. This restores managed API connector context so the designer renders connections correctly.

6. Smoke test the map

Open Data Mapper, load an existing map, save once if the extension regenerates XSLT, then run a local workflow test before pushing changes.

Pull request checks

Map review

Review the .lml design intent and the generated .xslt together. A visual save can regenerate runtime code.

Artifact presence

Confirm schemas, map definitions, maps, workflow folders, host.json, and connections.json are present before packaging.
