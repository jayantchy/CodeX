Setup

Make local projects portable

A team setup should let any developer clone the repo, restore local settings, open the Logic App Standard project root, and edit maps without machine-specific drift.

Developer machine baseline

1. Pin tools

Agree on VS Code, Azure Logic Apps Standard extension, Azure Functions Core Tools, and Node/runtime versions for the team.

2. Open the root

Open the Standard logic app project root, where host.json, connections.json, and workflow folders live.

3. Restore local settings

Each developer creates their own root local.settings.json and, when needed, the matching workflow-designtime/local.settings.json.

Local settings template

Use the same shape for the project root and the workflow-designtime folder. Keep real subscription, resource group, storage, and connector values local to each developer.

The JSON below is valid. The designer copy needs the same comma-separated properties; missing commas will stop the local runtime from reading the file.
{
  "IsEncrypted": false,
  "Values": {
    "AzureWebJobsStorage": "UseDevelopmentStorage=true",
    "AzureWebJobsFeatureFlag": "EnableMultipleLanguageWorker",
    "FUNCTIONS_EXTENSION_VERSION": "~4",
    "FUNCTIONS_WORKER_RUNTIME": "dotnet",
    "WORKFLOWS_SUBSCRIPTION_ID": "",
    "WORKFLOWS_RESOURCE_GROUP_NAME": "",
    "WEBSITE_NODE_DEFAULT_VERSION": "~18",
    "APP_KIND": "workflowapp"
  }
}

Commit and ignore rules

Commit the artifacts that define the map and workflow. Ignore files that are secrets, machine state, or per-user editor preferences.

CommitIgnore
Artifacts/MapDefinitions/*.lmllocal.settings.json
Artifacts/Maps/*.xslt**/local.settings.json
Artifacts/Schemas/*.vscode/settings.json if personal
workflow.json, connections.json, host.jsonRaw keys, connection strings, and generated clutter

Recommended handoff rule

Commit a sanitized local.settings.sample.json with placeholders, then ask each developer to create their real local files from that template.

For Data Mapper work, also commit Artifacts/MapDefinitions, Artifacts/Maps, Artifacts/Schemas, and any Artifacts/DataMapper/Extensions files.

Transfer a local workspace to another developer

1. Push the project root

Source control should include workflow folders, host.json, connections.json, Artifacts, and shared VS Code recommendations such as .vscode/extensions.json.

2. Recreate local settings

The receiving developer creates root and workflow-designtime local settings from the sample file, then fills subscription, resource group, storage, and connector values.

3. Open the correct folder

Open the folder that contains host.json. Opening a workflow subfolder or Artifacts folder can make the designer fail to recognize the project.

Transfer into a Standard Logic App

From Azure to local

In the Azure portal, open the Standard Logic App, use Download app content, then open the downloaded project or workspace in VS Code.

From local to Azure

Deploy the Standard logic app project to an existing or new Standard Logic App resource. The package must place workflow folders, host.json, and connections.json at the zip root.

After deployment

Update app settings, connection references, managed API connections, and access policies for the target environment. Local connection metadata is not automatically production-ready.

Pull request checks

Map review

Review the .lml design intent and the generated .xslt together. A visual save can regenerate runtime code.

Artifact presence

Confirm schemas, map definitions, maps, workflow folders, host.json, and connections.json are present before packaging.

Local setup guidance is based on Logic Apps Standard project and DevOps deployment documentation.
