Data Mapper functions, tips & troubleshooting

Master the key extensibility functions, learn practical tips from real projects, and troubleshoot common issues before they cost you hours.

Core extensibility functions

Data Mapper provides four key extensibility mechanisms. Each one solves a different problem — knowing which one to reach for saves significant debugging time.

⚙️

Run XSLT

Inject handcrafted XSLT into a visual map

Use Run XSLT when the visual designer cannot express the logic cleanly — complex grouping, recursion, legacy BizTalk migration, or advanced conditional branching.

Place your XSLT file at:

Artifacts/DataMapper/Extensions/InlineXSLT/my-transform.xslt

Then in the Data Mapper Functions pane, search for Run XSLT, add it to the surface, and select your file from the dropdown. Connect the output to your destination node.

<xsl:stylesheet version="3.0"
  xmlns:xsl="http://www.w3.org/1999/XSL/Transform">

  <!-- Group items by category and sum quantities -->
  <xsl:template match="/">
    <Summary>
      <xsl:for-each-group select="//Item"
        group-by="Category">
        <CategoryTotal>
          <Name><xsl:value-of select="current-grouping-key()"/></Name>
          <Total><xsl:value-of select="sum(current-group()/Qty)"/></Total>
        </CategoryTotal>
      </xsl:for-each-group>
    </Summary>
  </xsl:template>

</xsl:stylesheet>
When: Complex grouping, recursion, legacy map migration, or any logic that would create a tangled visual chain.
🔍

Execute XPath

Extract values with custom XPath 3.1 expressions

Use Execute XPath when you need precise value extraction from deeply nested structures, or when the visual Filter function produces unreliable results with string comparisons.

In the Functions pane → Utility category → Execute XPath. Enter your expression in the property box:

<!-- Extract the first item's price -->
/ns0:Order/Items/Item[1]/Price

<!-- Filter items by string value (reliable alternative to Filter) -->
/ns0:Order/Items/Item[Category='Electronics']/Name

<!-- Sum all quantities where price exceeds threshold -->
sum(/ns0:Order/Items/Item[Price > 50]/Qty)

<!-- Conditional value with XPath 3.1 -->
if (//Status = 'Active') then 'APPROVED' else 'PENDING'
When: Deep nested extraction, string-based filtering (workaround for Filter function bugs), conditional values, or aggregation.
📦

Custom XML Functions

Reusable XSLT 3.0 functions in the toolbox

Create reusable transformation logic that appears as a draggable function in the Data Mapper designer. Best for logic you use across multiple maps — date formatting, ID normalization, unit conversion.

Place the XML file at:

Artifacts/DataMapper/Extensions/Functions/my-functions.xml

Example: ISO date formatter and price calculator

<?xml version="1.0" encoding="utf-8"?>
<xs:schema xmlns:xs="http://www.w3.org/2001/XMLSchema"
  xmlns:dm="http://schemas.microsoft.com/azure/logic-apps/data-mapper">

  <!-- Format YYYYMMDD → ISO date -->
  <xsl:function name="dm:iso-date" as="xs:string"
    xmlns:xsl="http://www.w3.org/1999/XSL/Transform">
    <xsl:param name="dateText" as="xs:string" />
    <xsl:sequence select="
      concat(
        substring($dateText, 1, 4), '-',
        substring($dateText, 5, 2), '-',
        substring($dateText, 7, 2)
      )
    " />
  </xsl:function>

  <!-- Calculate line total with rounding -->
  <xsl:function name="dm:line-total" as="xs:decimal"
    xmlns:xsl="http://www.w3.org/1999/XSL/Transform">
    <xsl:param name="qty" as="xs:decimal" />
    <xsl:param name="price" as="xs:decimal" />
    <xsl:sequence select="round($qty * $price * 100) div 100" />
  </xsl:function>

</xs:schema>
When: Reusable logic across maps — date formatting, ID generation, currency conversion, or any transformation you call more than once.
🔄

xsl:apply-templates

Template-driven recursive transformation

xsl:apply-templates uses pattern matching to process XML sub-trees recursively. Unlike xsl:call-template (which works like a function call), apply-templates matches nodes by pattern and processes them automatically.

Logic Apps runs on the Saxon.NET XSLT 3.0 engine, so full template matching is supported. Use it via Run XSLT:

<xsl:stylesheet version="3.0"
  xmlns:xsl="http://www.w3.org/1999/XSL/Transform">

  <!-- Entry point: match root and apply to children -->
  <xsl:template match="/">
    <ProcessedOrders>
      <xsl:apply-templates select="Orders/Order" />
    </ProcessedOrders>
  </xsl:template>

  <!-- Transform each Order element -->
  <xsl:template match="Order">
    <ProcessedOrder>
      <ID><xsl:value-of select="OrderID" /></ID>
      <Customer><xsl:value-of select="BuyerName" /></Customer>
      <Status>Processed</Status>
      <Items>
        <xsl:apply-templates select="LineItems/Item" />
      </Items>
    </ProcessedOrder>
  </xsl:template>

  <!-- Transform each Item within an Order -->
  <xsl:template match="Item">
    <Line>
      <SKU><xsl:value-of select="ProductCode" /></SKU>
      <Qty><xsl:value-of select="Quantity" /></Qty>
      <Total><xsl:value-of select="Quantity * UnitPrice" /></Total>
    </Line>
  </xsl:template>

</xsl:stylesheet>
apply-templates vs call-template: Use apply-templates when you want the XSLT engine to decide which template to invoke based on the node type. Use call-template when you want to explicitly invoke a named template like a function call.
When: Recursive XML processing, polymorphic transforms (different templates for different node types), or processing deeply nested hierarchies where explicit visual mapping becomes unmanageable.

Tips & tricks from real projects

Use built-in functions first

Data Mapper has 100+ built-in functions (string, math, date, collection). Reach for custom XSLT only after verifying no built-in covers the case. Built-ins integrate with the visual debugger; custom code doesn't.

Enforce strict typing

The mapper may coerce 12345 to 12345.0 in JSON output. Use toInteger() or number() to enforce the output type before downstream APIs reject it silently.

Keep schema basenames unique

Don't place Customer.json and Customer.xsd in the same folder. Same-basename collisions can cause the wrong schema to load. Use descriptive prefixes: src_Customer.xsd, tgt_Customer.json.

Save before every test run

The local test panel reads the generated .xslt file. If you changed the visual map but didn't save, tests run against the old XSLT. Always Ctrl+S before clicking Test.

Edit the .lml directly for bulk changes

The .lml file is YAML-based. For bulk renames or repetitive connection edits, editing the YAML directly can be faster than the visual surface. Always reopen the map to verify after manual edits.

Store test payloads in source control

Create a tests/mapsamples/ folder with representative JSON/XML inputs and expected outputs. This makes regressions visible in PRs, even though the mapper has no native multi-payload test chooser yet.

Use Execute XPath instead of Filter for strings

The visual Filter function has documented bugs with string comparisons (#6205). Use Execute XPath with predicates like Item[name='Pen'] for reliable filtering.

Modularize complex maps

If a single map canvas becomes crowded and hard to maintain, split the transformation into sequential steps — use multiple smaller maps chained in the workflow rather than one monolithic map.

Review generated XSLT in PRs

Always review both the .lml (design intent) and the .xslt (runtime behavior) together. A visual save can regenerate XSLT in unexpected ways — especially for Filter and repeating elements.

Troubleshooting guide

The most common issues and their fixes, sourced from Microsoft docs, GitHub issues, and community reports.

⚠️

"Undefined" errors when calling a map

Check local.settings.json: set FUNCTIONS_WORKER_RUNTIME to dotnet-isolated and add AzureWebJobsFeatureFlags: EnableMultiLanguageWorker.

⚠️

Filter function returns wrong results

String comparisons in Filter are unreliable (#6205). Switch to Execute XPath with item[name='value'] predicates.

⚠️

Number formatting — 12345 becomes 12345.0

JSON numbers can be coerced to floating point. Wrap with toInteger() or use a string cast if the downstream API expects exact integer format.

⚠️

Schema fails to load — "$ref" not resolved

JSON schemas using $ref can fail to load (#6924). Inline the referenced definitions or flatten the schema until the fix ships.

⚠️

"Not a logic app project" on fresh clone

Open the project root folder (where host.json lives), not a subfolder. Pin the VS Code extension version (#7623).

⚠️

Map works in editor but fails in workflow

Verify the input payload matches the schema exactly. Check that you're using Transform using Data Mapper XSLT (not Transform XML). Add a Compose step as a workaround for action-shape quirks.

✓

XSLT Debugger for local inspection

Install the XSLT Debugger extension for VS Code. Set breakpoints, inspect variables, and watch XPath expressions in the generated XSLT — much more powerful than the built-in Test Map panel.

✓

Use DataMapTestExecutor for CI testing

The Logic Apps Standard testing SDK includes DataMapTestExecutor for automated map validation. Run map tests in your CI pipeline to catch regressions before deployment.
